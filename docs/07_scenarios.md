# 07. 역할별 기본값과 사용 시나리오

실제 화면을 보기 전에, 검토자와 개발자가 각각 어떤 흐름으로 쓰게 되는지 미리 그려본다.

## 7.1 역할 한 번 정해두면 어떻게 기억되나

[03_ui_design.md](03_ui_design.md) 3.2의 사이드바 위젯에서 이름 + 역할을 한 번 저장하면 세 단계로 기억된다.

1. **세션 동안** — `st.session_state["user"] = {"name": ..., "role": ...}` (브라우저 탭이 유지되는 동안)
2. **브라우저 단위 보관** — `extra-streamlit-components`의 `CookieManager`로 쿠키에도 동일 정보 저장 → 다음 방문 시 자동 로드
3. **모든 페이지에서 참조** — `app.py` 진입부에서 한 번 로드해 페이지 분기에 사용

```python
# ui/auth.py 핵심 (쿠키 포함 버전)
import extra_streamlit_components as stx
import streamlit as st

def get_or_init_user():
    cookie_mgr = stx.CookieManager(key="user_cookie")
    saved = cookie_mgr.get("user")  # {"name", "role"} 또는 None

    if "user" not in st.session_state and saved:
        st.session_state["user"] = saved

    user = st.session_state.get("user")
    if not user or not user.get("name"):
        with st.sidebar:
            name = st.text_input("이름")
            role = st.radio("역할", ["reviewer", "developer"],
                            format_func=lambda r: "검토자" if r == "reviewer" else "개발자")
            if st.button("저장") and name:
                st.session_state["user"] = {"name": name, "role": role}
                cookie_mgr.set("user", st.session_state["user"], expires_at=...)
                st.rerun()
    return st.session_state.get("user")
```

공용 PC에서 같은 브라우저를 여러 사람이 사용하는 환경이면 매번 "변경"을 눌러야 하므로, 사이드바 상단에 **"현재: 김OO (검토자) [변경]"** 표시를 항상 노출한다.

## 7.2 역할별 기본값 차이 요약

| 항목 | 검토자 (reviewer) | 개발자 (developer) |
|---|---|---|
| 대시보드 첫 강조 영역 | 검토 대기 + 내가 등록한 미해결 | 처리 큐 (요청됨/재요청/내게 할당) |
| 사이드바 액션 큐 라벨 | "검토 대기 N건" | "처리 대기 N건" |
| 가장 큰 버튼 (메인 CTA) | [+ 새 요청 등록] | [내 큐 보기] |
| 요청목록 진입 시 기본 필터 | `author = me AND status ≠ closed` | `assignee ∈ {me, 미할당} AND status ∈ 활성` |
| 상세 페이지 액션 영역 | 검토 완료 / 재요청 / 즉시 클로즈 | 확인중 / API대기 / 완료 |
| 알림 카운트 산정 기준 | `status = done AND author = me` | `status ∈ {requested, reopened} AND assignee = me` |
| 새 요청 등록 페이지 진입성 | 메인 CTA로 강조 | 메뉴에서만 (드물게 사용) |

기본 필터는 "고정"이 아니라 "초기값" — 사용자가 필터를 변경하면 즉시 반영되며 다음 방문 시에도 마지막 선택을 기억(쿠키)할 수 있음.

## 7.3 검토자 첫 화면

```
┌─────────────┬──────────────────────────────────────────────────┐
│ Daily View  │ 안녕하세요, 김OO님 (검토자)                       │
│             │                                                  │
│ 현재         │ [+ 새 요청 등록]                ← 메인 CTA, 큰 버튼│
│ 김OO        │ ──────────────────────────────────────────────── │
│ (검토자)    │ ▣ 검토 대기 (3)                  ← 최상단 강조    │
│ [변경]      │   #..._a3f1b2  로그인 500 에러   [긴급] 완료 5분전│
│             │   #..._b9d2c1  차트 색상 표준화  [보통] 완료 1시간│
├─────────────┤   #..._c4e8a0  메뉴 정렬 오류    [보통] 완료 어제 │
│ 검토 대기 3건│                                                  │
│ (긴급 1)    │ ▣ 내가 등록한 미해결 (8)                          │
│             │   #..._d5f2e1  팝업 X버튼 동작   [긴급] 확인중    │
│ 메뉴        │   #..._e8a3b2  비밀번호 변경     [보통] API대기   │
│ 대시보드    │   ...                                             │
│ 요청목록    │                                                  │
│ + 새 요청 등록│ ▣ 전체 현황                                       │
│ 통계        │   긴급 3 / 보통 12 / 낮음 4                       │
│             │   요청 5 · 확인중 8 · 완료 6 · 검토중 3            │
└─────────────┴──────────────────────────────────────────────────┘
```

검토자는 "내가 보낸 일이 처리되어 돌아왔는지"가 가장 큰 관심사 → **검토 대기**가 최상단. 새 버그 발견 시 빠르게 등록할 수 있게 **+ 새 요청 등록**이 큰 CTA.

## 7.4 개발자 첫 화면

```
┌─────────────┬──────────────────────────────────────────────────┐
│ Daily View  │ 안녕하세요, 이OO님 (개발자)                       │
│             │                                                  │
│ 현재         │ [내 큐 전체 보기]                ← 메인 CTA      │
│ 이OO        │ ──────────────────────────────────────────────── │
│ (개발자)    │ ▣ 처리 큐 (6)                  ← 최상단, 빨간 띠 │
│ [변경]      │   ⚠ #..._a3f1b2  로그인 500 에러 [긴급] 요청 30분 │
│             │     #..._d5f2e1  팝업 X버튼 동작 [긴급] 재요청 어제│
├─────────────┤     #..._c4e8a0  메뉴 정렬 오류  [보통] 요청 1일전│
│ 처리 대기 6건│     ...                                          │
│ (긴급 2)    │                                                  │
│             │ ▣ 외부 대기 중 (2)                ← 장기 추적용  │
│ 메뉴         │   #..._e8a3b2  비밀번호 변경     [보통] API 5일  │
│ 대시보드     │                                                  │
│ 요청목록     │ ▣ 최근 내 활동 (5)                                │
│ 새 요청 등록 │   • 어제  #..._x... 에 코멘트                     │
│ 통계        │   • 어제  #..._y... 상태 → 완료                   │
└─────────────┴──────────────────────────────────────────────────┘
```

개발자는 "오늘 뭘 해야 하지"가 가장 큰 관심사 → **처리 큐**가 최상단, SLA 임박은 빨간 띠. **내 큐 전체 보기**가 메인 CTA. 새 요청 등록은 메뉴에는 있지만 강조하지 않음 (개발자가 등록하는 경우는 드뭄).

## 7.5 시나리오 A: 검토자가 새 버그 발견 → 등록

**상황**: 김OO가 프로그램 사용 중 로그인 화면에서 500 에러 발견.

| 단계 | 행동 | 화면 변화 |
|---|---|---|
| 1 | Win+Shift+S 로 화면 캡처 | 클립보드/png |
| 2 | Daily View 접속 (쿠키로 자동 식별) | 검토자 대시보드 |
| 3 | [+ 새 요청 등록] 클릭 | 등록 폼 페이지 |
| 4 | 제목/설명/긴급도=긴급 입력 | 폼 입력 |
| 5 | 스크린샷 드래그&드롭 | 미리보기 표시 |
| 6 | 담당 개발자=이OO 선택 (선택) | 드롭다운 |
| 7 | [등록] 클릭 | 토스트: "등록되었습니다" |
| 8 | 자동 이동 → 요청목록 (내가 등록한 항목 필터) | 방금 올린 카드가 맨 위 [긴급][요청됨] |

이 시점 자동 기록 (사용자 입력 X):
- `created_at`, `updated_at` = 등록 시각
- `status = requested`, `status_history`에 첫 항목
- `audit.log`에 `create_issue` 라인

## 7.6 시나리오 B: 개발자가 아침에 출근해서 처리

**상황**: 이OO가 출근해서 어제부터 쌓인 항목 처리.

| 단계 | 행동 | 화면 변화 / 자동 기록 |
|---|---|---|
| 1 | Daily View 접속 | 개발자 대시보드 — 처리 큐 6건, SLA 임박 1건 빨간 띠 |
| 2 | 빨간 띠 카드 #..._a3f1b2 클릭 | 상세 페이지 |
| 3 | 스크린샷 + 설명 읽음 | — |
| 4 | 상태 드롭다운 → "확인중" 선택 | `in_progress`로 변경, 시스템 코멘트 자동 기록 |
| 5 | 코멘트 "확인 시작합니다, 오늘 안에 답변" → [등록] | 타임라인에 코멘트 추가, `updated_at` 갱신 |
| 6 | (실제 코드 작업) | 앱 외부 |
| 7 | 다시 접속, 같은 항목 상세 | — |
| 8 | 상태 → "완료", 코멘트 "수정 적용, 확인 부탁드립니다" | `done` 변경, 검토자의 검토대기 큐로 자동 이동 |

## 7.7 시나리오 C: 외부 API 의존 발견

**상황**: 작업 중 우리 코드 문제가 아닌 외부 API 응답 형식 변경 확인.

1. 상세 페이지에서 상태 → "API대기" 선택 → 시스템 코멘트 자동 기록
2. 코멘트 작성: "API 응답 형식이 예상과 다름. 외부 팀에 문의 보냈음. 답변 받으면 진행."
3. 검토자도 이 변경을 즉시 볼 수 있음 (자동 새로고침 30초)
4. "외부 대기 중" 칩 필터로 모든 API대기 항목 추적 가능
5. 5일 경과 시 카드에 경고 배지 자동 표시 ([04_workflow.md](04_workflow.md) 4.7 자동화 규칙)

## 7.8 시나리오 D: 검토자가 완료 항목 재확인

**상황**: 김OO가 이OO의 "완료" 처리를 검증.

| 단계 | 행동 | 결과 / 자동 기록 |
|---|---|---|
| 1 | 검토자 대시보드 진입 | "검토 대기 (3)" 최상단에 항목 표시 |
| 2 | 항목 카드 클릭 | 상세 페이지 |
| 3 | 개발자 코멘트 + 스크린샷 확인 | — |
| 4 | 실제 프로그램에서 동작 검증 | 앱 외부 |
| 5a | 정상 → [검토 완료] 클릭 | `status = closed`, `reviewer_confirmed = true`, `reviewer_confirmed_at` 자동 기록 |
| 5b | 여전히 문제 → [재요청] + 코멘트 + 새 스크린샷 첨부 | `status = reopened`, 개발자 처리 큐로 다시 이동 |

## 7.9 시나리오 E: 재요청 → 재처리 사이클

**상황**: 검토자가 재요청 보냈고, 개발자가 다시 처리.

1. 개발자 대시보드 → 처리 큐에 [재요청] 배지 항목 표시 (빨간 강조)
2. 상세 진입 → 검토자가 추가한 새 스크린샷 + 코멘트 확인
3. 상태 → "확인중"
4. 코멘트 "다시 확인. 이번엔 캐시 무효화 함께 적용."
5. 작업 후 상태 → "완료"
6. 검토자는 7.8 시나리오로 다시 진입

## 7.10 시나리오 F: 코멘트만 주고받는 경우 (상태 변경 없음)

**상황**: 개발자가 작업 중 추가 정보가 필요해 검토자에게 질문.

1. 상세 페이지에서 코멘트 "어떤 브라우저에서 재현되는지 알 수 있을까요?" → [등록]
2. 검토자 대시보드 알림 카운트는 변하지 않음 (상태가 `in_progress` 그대로)
3. 다만 검토자가 그 항목 상세에 들어가면 새 코멘트가 보임
4. 검토자: "Chrome 120, Edge 119에서 재현됩니다" 코멘트
5. 개발자가 다시 들어오면 새 코멘트 확인 → 작업 계속

> 코멘트만으로 상태가 바뀌지 않는다는 점이 포인트. 알림은 "내가 액션해야 할 때"에만 카운트되도록 단순화 ([04_workflow.md](04_workflow.md) 4.5).

## 7.11 직접 보고 싶다면 — 프로토타입 옵션

여기까지는 설계만 한 상태. 실제 화면 감을 빠르게 잡고 싶다면 두 옵션이 있다.

### 옵션 1: UI만 있는 모의 프로토타입 (빠름)
- 진짜 저장은 안 하고, 더미 항목 5~10개를 하드코딩해서 클릭 가능한 화면만 만든다
- 등록/상태변경 버튼은 메모리상 토글만 됨 (페이지 새로고침하면 초기화)
- 작업량: 1~2시간
- 목적: "이런 모양인지" 한 번 확인 후 본 구현 진입
- 단점: 이 코드는 본 구현 때 거의 다 버려짐

### 옵션 2: M0 + M1 일부 + M2 진입 (권장)
- [06_implementation_plan.md](06_implementation_plan.md) 의 정상 순서
- 코어(저장소) 먼저 만들고 UI 입힘 → 첫 화면이 뜨면 그게 곧 본 구현
- 작업량: 검토자/개발자 둘 다 쓸 만한 최소 화면까지 약 4~5일
- 단점: 시간이 좀 걸림

**판단 기준**: "이 모양이 맞나"부터 확인해야 마음이 놓이면 옵션 1, "어차피 만들 건데 빨리 본 구현 들어가자" 면 옵션 2.

옵션 1이 필요하면 알려달라. 그 경우 별도 폴더(`prototype/`)에 격리해서 만들어두고, 본 구현은 옵션 2 흐름대로 진행한다.
