# 06. 구현 계획

## 6.1 개요

기능을 한꺼번에 만들지 말고, **MVP → 안정화 → 편의 기능** 순으로 구현.
각 마일스톤은 그 자체로 사용 가능한 형태로 끝나야 함.

## 6.2 마일스톤

### M0: 환경 구축 (반나절)

- [ ] Python 3.12.7 설치 확인
- [ ] 프로젝트 루트에 `.venv` 생성
- [ ] `requirements.txt` 작성 후 `pip install`
- [ ] `.env.example`, `.gitignore` 작성
- [ ] `streamlit hello` 가 뜨는 것 확인

### M1: 코어 데이터 계층 (1~2일)

UI 없이 파이썬 코드만으로 항목 생성/조회/상태변경/코멘트가 동작해야 함. CLI나 pytest로 검증.

- [ ] `core/models.py` — pydantic 모델 (`Issue`, `Comment`, `StatusEvent`, `ImageRef`)
- [ ] `core/locking.py` — `atomic_write_json()`, `FileLock` 헬퍼
- [ ] `core/repository.py` — `create_issue()`, `get_issue()`, `list_issues()`, `update_status()`, `add_comment()`, `add_image()`
- [ ] `core/index.py` — `update_index_entry()`, `rebuild_index()`, `read_index()`
- [ ] `core/workflow.py` — `allowed_transitions(current, role)`, `apply_transition()`
- [ ] `core/logger.py` — `audit_log(actor, action, item_id, detail)`
- [ ] `tests/test_repository.py` — 기본 동작 + 동시 쓰기 시나리오

### M2: 최소 UI (2~3일)

검토자/개발자가 실제로 사용할 수 있는 수준의 페이지들.

- [ ] `app.py` — 진입점, 사이드바 사용자 식별, 대시보드 카운트
- [ ] `ui/auth.py` — 사이드바 사용자 위젯
- [ ] `ui/theme.py` — 색상 상수
- [ ] `ui/components.py` — 카드, 배지, 긴급도 라벨
- [ ] `pages/2_새요청등록.py` — 폼 + 다중 이미지 업로드
- [ ] `pages/1_요청목록.py` — 필터(긴급도/상태/담당자/검색) + 카드 그리드 + 페이지네이션
- [ ] `pages/3_상세보기.py` — 이미지 갤러리, 상태 변경, 코멘트 타임라인 + 입력

여기까지 하면 PPT 대체 사용 가능.

### M3: 운영 안정화 (1~2일)

- [ ] 동시성 검증 — 두 브라우저로 같은 항목 편집 시 충돌 처리
- [ ] index.json 정합성 검증 — 앱 시작 시 자동 점검
- [ ] 자동 새로고침 (`streamlit-autorefresh`) — 30초 간격
- [ ] 입력 검증 (제목 공백, 이미지 크기/장수 제한)
- [ ] 한글 파일명 슬러그화
- [ ] 백업 스크립트 + 작업 스케줄러 등록
- [ ] NSSM으로 윈도우 서비스 등록
- [ ] 사내 네트워크에서 실제 접속 확인

### M4: 편의 기능 (선택, 우선순위별)

- [ ] `pages/4_통계.py` — 긴급도별/상태별 차트, SLA 위반 카운트
- [ ] 사전 정의 필터 뷰 (스마트 큐) — 칩 형태로 노출
- [ ] 자동 아카이브 (closed → 14일 경과)
- [ ] SLA 임박 표시 (카드 빨간 테두리)
- [ ] 인용 답장 (코멘트)
- [ ] 쿠키로 사용자 정보 보존 (`extra-streamlit-components`)
- [ ] CSV 내보내기 (필터된 목록)
- [ ] 검토 완료 후 24시간 내 코멘트 → 자동 재오픈

### M5: 추후 확장 (필요 시점에)

- [ ] 이메일/슬랙 알림 어댑터
- [ ] 권한 강화 (단순 역할 → 사용자 화이트리스트)
- [ ] SQLite 마이그레이션 (1만 건 넘어갈 때)
- [ ] 모바일 친화 레이아웃 (Streamlit 한계 → 별도 검토)

## 6.3 우선순위 결정 원칙

| 기능 | M레벨 | 기준 |
|---|---|---|
| 핵심 워크플로우 (등록/상태변경/코멘트/검토완료) | M1+M2 | 없으면 사용 불가 |
| 이미지 업로드 | M2 | PPT의 핵심 기능 대체 |
| 필터/검색 | M2 | 항목 누적 시 필수 |
| 동시성 안정성 | M3 | 데이터 손실 위험 직결 |
| 자동 백업 | M3 | 사고 시 복구 |
| 통계/차트 | M4 | 있으면 좋지만 없어도 됨 |
| 알림 | M4~M5 | 대시보드 카운트로 일단 대체 |

"있으면 좋은" 기능은 M2가 실제로 사용되기 전엔 만들지 않는다.

## 6.4 검증 체크리스트 (M3 완료 시)

운영 시작 전에 반드시 확인:

- [ ] 두 사용자가 동시에 새 항목을 등록해도 모두 저장되는가
- [ ] 두 사용자가 같은 항목에 동시에 코멘트를 달 때 둘 다 보존되는가
- [ ] 한 사용자가 상태를 변경하는 동안 다른 사용자가 같은 항목의 상태를 변경하면 어떻게 되는가 (마지막 쓰기 우선 + 경고)
- [ ] 앱을 강제 종료한 직후 재시작했을 때 인덱스가 자동 복구되는가
- [ ] 데이터 폴더만 백업하고 다른 PC에 복원했을 때 정상 동작하는가
- [ ] 한글 제목/설명/파일명 모두 정상 저장/표시되는가
- [ ] 10MB 이미지 업로드가 정상 동작하는가
- [ ] 1년치(2,500건) 더미 데이터에서 목록 페이지 응답 시간이 1초 이내인가
- [ ] 같은 네트워크의 다른 PC에서 정상 접속되는가
- [ ] Windows 재부팅 후 NSSM 서비스가 자동 시작되는가

## 6.6 위험 요소 및 대응

| 위험 | 영향 | 대응 |
|---|---|---|
| Streamlit의 rerun 모델로 인한 UX 어색함 | 보통 | 폼 + 콜백 패턴 적극 활용, session_state 캐시 |
| 다중 사용자 동시 편집 시 충돌 | 큼 | filelock + atomic write, 마지막 갱신 비교로 감지 |
| 데이터 손상 (전원 차단 등) | 큼 | atomic write로 부분 쓰기 방지, 매일 백업 |
| 항목 누적으로 인덱스 비대화 | 작음 | 1년 ≈ 1MB 미만, 실제 임계는 1만 건 (그때 SQLite) |
| 한글/유니코드 인코딩 이슈 | 작음 | 모든 I/O `encoding="utf-8"` 명시, 파일명 슬러그화 |
| 공용 PC 재부팅 후 서비스 미시작 | 보통 | NSSM 등록, 시작 후 healthcheck 자동화 |
