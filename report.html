작성일: 2026-06-03 운영 배포 직전 · HTTPS 설정 대기 대상: 사내 검토자·개발자 소수 인원
1. 사업 배경 및 목적

현재 개발 요청은 PowerPoint에 스크린샷을 붙이고 텍스트 코멘트와 긴급도(상·중·하)를 기재하는 방식으로 운영되고 있다. 이 방식은 (1) 누적 항목 정리가 어렵고, (2) 누가 어떤 항목을 처리 중인지 한눈에 파악되지 않으며, (3) 긴급도·상태별 필터·검색이 불가능하다. 본 프로젝트는 이를 로컬 호스팅 웹앱으로 전환하여, 사내 공용 PC를 통해 검토자와 개발자가 동일한 화면에서 요청·답변·검토를 체계적으로 주고받도록 한다. DB 없이 파일 기반으로 운영하여 인프라 부담을 최소화한다.

4. 기술 구성

플랫폼: Python 3.12.7 Streamlit ≥ 1.39 Windows 공용 PC 호스팅 DB 미사용(파일 기반)

주요 라이브러리: pydantic Pillow filelock pandas streamlit-paste-button streamlit-autorefresh tzdata

저장 구조: 항목별 폴더(data/items/{ID}/)에 meta.json, comments.jsonl, 이미지를 격리 저장. 루트 index.json이 목록 캐시 역할. 모든 쓰기는 filelock + atomic write 패턴으로 다중 사용자 동시 접속 시 충돌을 차단한다.

5. 주요 리스크 및 미해결 사항
[A · 치명] HTTPS 미설정 → 사내 다른 PC에서 클립보드 붙여넣기 불가
브라우저 보안 정책상 클립보드 API는 HTTPS 또는 localhost 환경에서만 동작한다. 현재 RUNBOOK은 평문 HTTP 기반 NSSM 등록 절차만 포함하고 있어, 검토자가 자기 PC에서 사내 IP로 접속할 경우 핵심 워크플로(캡처→붙여넣기→등록)가 차단된다. 클립보드 붙여넣기는 사용자가 명시한 필수 기능이므로 배포 전 반드시 해소되어야 한다.
대응안: mkcert로 사내용 root CA 발급 → Streamlit sslCertFile/sslKeyFile 적용 → 각 클라이언트 PC에 root CA 1회 import. (절차는 본 보고서 직전 세션 정리에 존재)
[B · 중요] 사내 공용 PC 실배포 미진행
코드와 운영 매뉴얼(RUNBOOK.md)은 완비되었으나 실제 NSSM 서비스 등록, 방화벽 8501 인바운드 허용, 일일 백업 작업 스케줄러 등록이 수행되지 않았다. 위 항목 A의 HTTPS 작업과 함께 일괄 실행하는 것이 효율적이다.
[C · 일반] 실사용자 피드백 미수집
검토자 1명 + 개발자 1명 대상 1주 파일럿 운영이 계획되어 있으나 미실시. 실사용 데이터가 없으면 M4 우선순위(통계 고도화·스마트 필터 칩·자동 아카이브 강화 등) 판단이 어렵다.
6. 다음 단계 (우선순위 순)
  1. HTTPS 셋업. mkcert 설치 → 사내 IP·호스트명 인증서 발급 → .streamlit/config.toml에 SSL 경로 반영 → RUNBOOK 5장에 절차 정식 추가.
  2. 사내 공용 PC 배포. NSSM 서비스 등록, 방화벽 인바운드 허용, 작업 스케줄러에 일일 백업 등록.
  3. 클라이언트 PC 사전 작업. 검토자·개발자 PC 각각에 root CA 1회 import(PowerShell 한 줄).
  4. 1주 파일럿 운영. 검토자 1명 + 개발자 1명 실사용 진행, 사용 빈도·불편 지점·누락 기능 수집.
  5. M4 진입 여부 결정. 피드백 기반으로 통계 고도화·스마트 필터·자동 아카이브 강화 항목을 선별 적용.
